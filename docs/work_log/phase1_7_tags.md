## 결정과 이유

- **태그 하나만 (상태 전이 모델):** 공고당 태그 1개만 가능. DB unique 제약 `(tenant_id, notice_type, notice_id)` 그대로 활용. 검토요청→입찰대상→낙찰/유찰 순으로 상태가 전이되는 것이 자연스러움.
- **PUT upsert 방식:** 공고당 태그 1개이므로 POST(create) + PATCH(update) 대신 PUT 하나로 생성/변경 처리. 프론트에서 태그 버튼 클릭 시 동일 API 호출.
- **같은 태그 재클릭 → 삭제:** UX 편의성. 태그 해제를 위해 별도 "삭제" 버튼 없이 토글 방식.
- **태그는 테넌트(회사) 단위 공유:** 같은 회사 내에서 공고에 대한 판단이 하나로 수렴되는 것이 자연스러움. tagged_by로 마지막 변경자 기록.
- **_build_tag_map() 일괄 조회:** 공고 목록에서 N+1 쿼리 방지. 현재 페이지 공고 ID를 모아서 한 번에 태그 조회.
- **TenantTag 모델 신규 생성 불필요:** 마이그레이션 001에서 이미 tenant_tags 테이블 생성됨. 스키마+라우터만 추가.
- **TAG_COLORS 프론트에서 관리:** 백엔드는 태그 값(문자열)만 저장, 색상은 프론트 상수로 관리.
- **검토요청 전용 페이지(/review):** 공고 목록의 태그 필터로도 볼 수 있지만, 검토요청은 자주 쓰는 핵심 워크플로우이므로 사이드바에 전용 메뉴 추가. `useNotices({ tag: "검토요청" })` 호출로 별도 API 없이 기존 엔드포인트 재사용.
